## 1. 流程概述

### 1.1 设计原则

Deep Research 技能的执行流程围绕三大核心设计原则构建：

#### 全面性 (Comprehensiveness)

- 从多个来源收集信息，确保研究的全面性
- 覆盖多个维度的分析，提供多角度的洞察
- 综合不同观点，识别共识和争议

#### 准确性 (Accuracy)

- 验证信息来源的可信度和权威性
- 确保引用的准确性和完整性
- 对分析结果进行验证和交叉检查

#### 效率性 (Efficiency)

- 优化信息收集和分析过程，减少响应时间
- 采用并行处理和缓存机制，提高执行效率
- 提供不同详细程度的输出选项，满足不同需求

## 2. 主执行流程

### Step 1: 参数解析与验证

```
┌─────────────┐
│  接收请求    │ ──→ 解析 JSON/YAML/自然语言参数
└─────────────┘        │
                       ↓
              ┌──────────────────┐
              │  参数完整性检查   │
              │  • 必填参数存在?   │──→ ❌ → 返回 E001-E009
              │  • 类型正确?      │──→ ❌ → 返回 E002
              │  • 值在范围内?     │──→ ⚠️ → 返回 E003 或修正
              │  • 无参数冲突?     │──→ ❌ → 返回 E004
              └────────┬─────────┘
                       ↓ ✅
              ┌──────────────────┐
              │  应用默认值       │
              │  构建内部配置对象  │
              └────────┬─────────┘
                       ↓
              [进入 Step 2]
```

#### 输入

**原始请求**：可以是以下任一形式

| 输入类型 | 格式示例 | 处理方式 |
|---------|---------|---------|
| 结构化 JSON | `{"topic": "人工智能", "detail_level": "standard"}` | 直接解析 |
| 自然语言 | "请研究人工智能的发展趋势" | NLP 提取参数 |
| 命令格式 | `/research 人工智能 detailed` | 映射为配置 |
| 混合形式 | "研究人工智能在医疗领域的应用，详细版" | 组合解析 |

#### 处理逻辑

1. **请求解析**：尝试解析结构化输入，如失败则使用 NLP 提取参数
2. **参数校验**：检查必填参数、类型、范围和逻辑一致性
3. **默认值应用**：为可选参数应用合理的默认值
4. **配置对象构建**：创建标准化的内部配置对象

#### 输出

| 输出类型 | 条件 | 说明 |
|---------|------|------|
| `InternalConfig` | 所有校验通过 | 标准化的内部配置对象 |
| `ErrorResponse` | 校验失败 | 包含错误码、错误描述、修复建议 |

**InternalConfig 结构**：

```typescript
interface InternalConfig {
  topic: string;                  // 研究主题
  detailLevel: 'summary' | 'standard' | 'detailed';  // 详细程度
  template: 'standard' | 'academic' | 'industry' | 'policy';  // 模板变体
  language: 'zh-CN' | 'en-US';    // 输出语言
  focusAreas: string[];           // 重点关注领域
  sourceTypes: string[];          // 来源类型
  timeRange: { start: string; end: string };  // 时间范围
  citationStyle: 'numbered' | 'author-year';  // 引用风格
  outputFormat: 'markdown' | 'json';  // 输出格式
  maxSources: number;             // 最大引用来源数量
  rawRequest: string;             // 保留原始请求用于追溯
  parsedAt: Date;                 // 解析时间戳
  validationWarnings?: string[];  // 非致命警告
}
```

#### 耗时预估

< 1 秒

### Step 2: 研究主题分析

```
┌──────────────────┐
│  主题分析        │
│                  │
│  • 主题理解      │ ← 识别核心概念和相关领域
│  • 范围界定      │ ← 确定研究边界和重点
│  • 关键词提取    │ ← 生成搜索关键词和查询语句
└────────┬─────────┘
         ↓
┌──────────────────┐
│  分析结果生成    │
│  • 子主题识别    │
│  • 相关领域映射  │
│  • 搜索策略制定  │
└────────┬─────────┘
         ↓
    [进入 Step 3]
```

#### 输入

`InternalConfig` （来自 Step 1 的标准化配置）

#### 处理逻辑

1. **主题理解**：分析研究主题，识别核心概念和相关领域
2. **范围界定**：确定研究的边界和重点，避免范围过大或过小
3. **关键词提取**：生成相关的搜索关键词和查询语句
4. **子主题识别**：将主题分解为多个子主题，便于后续分析
5. **搜索策略制定**：根据主题特点制定合适的信息收集策略

#### 输出

**TopicAnalysis（主题分析结果）**：

```typescript
interface TopicAnalysis {
  coreConcepts: string[];         // 核心概念
  subtopics: string[];            // 子主题
  relatedFields: string[];        // 相关领域
  searchKeywords: string[];       // 搜索关键词
  searchQueries: string[];        // 搜索查询语句
  estimatedComplexity: 'low' | 'medium' | 'high';  // 预估复杂度
  recommendedSources: string[];   // 推荐的来源类型
  timeEstimate: number;           // 预估执行时间（分钟）
}
```

#### 耗时预估

3-5 秒

### Step 3: 信息收集阶段

> **注意**：这是整个流程的核心阶段，也是最复杂的步骤，通常占总耗时的 40-50%。

```
                    ┌──────────────────────┐
                    │   Step 3: 信息收集    │
                    │                      │
   ┌────────────────┤  3.1 数据源适配      ├────────┐
   │                │  • 学术数据源        │        │
   │  数据源层      │  • 行业数据源        │        │
   │                │  • 新闻数据源        │        │
   └────────────────┤                      ├────────┤
                    │  3.2 信息检索         │        │
                    │  • 关键词搜索         │        │
   ┌────────────────┤  • 语义搜索           ├────────┤
   │                │  • 相关度排序         │        │
   │  检索层        │                      │        │
   └────────────────┤  3.3 数据过滤         ├────────┤
                    │  • 可信度评估         │        │
   ┌────────────────┤  • 相关性过滤         │        │
   │                │  • 重复内容去重       │        │
   │  过滤层        │                      │        │
   └────────────────┤  3.4 数据提取         ├────────┘
                    │  • 关键信息提取       │
                    │  • 引用信息提取       │
                    │  • 元数据提取         │
                    └──────────┬───────────┘
                               ↓
                   ┌────────────────────┐
                   │  CollectedData      │
                   │  (结构化数据集)      │
                   └────────────────────┘
```

#### 输入

`TopicAnalysis` （来自 Step 2 的主题分析结果）

#### 子步骤详解

##### 3.1 数据源适配

**学术数据源**：
- 学术论文数据库（如 arXiv、IEEE Xplore、ACM Digital Library）
- 学术搜索引擎（如 Google Scholar）
- 学术机构网站和知识库

**行业数据源**：
- 行业报告和白皮书
- 市场研究机构数据
- 行业协会和组织发布的信息

**新闻数据源**：
- 主流新闻媒体
- 专业领域新闻网站
- 财经和科技媒体

**官方数据源**：
- 政府网站和官方发布
- 监管机构文档
- 国际组织报告

**专家数据源**：
- 专家博客和专栏
- 专业论坛和社区
- 行业专家访谈

##### 3.2 信息检索

- **关键词搜索**：使用提取的关键词进行精确搜索
- **语义搜索**：使用自然语言处理技术进行语义相关搜索
- **相关度排序**：根据与主题的相关度对搜索结果进行排序
- **多维度检索**：从不同角度和维度进行检索，确保信息的全面性

##### 3.3 数据过滤

- **可信度评估**：评估信息来源的可信度和权威性
- **相关性过滤**：过滤与主题不相关的信息
- **重复内容去重**：去除重复或高度相似的内容
- **时效性检查**：优先选择最新的信息，特别是对于快速发展的领域

##### 3.4 数据提取

- **关键信息提取**：从原始文本中提取关键信息和要点
- **引用信息提取**：提取引用信息，包括作者、标题、来源、发布日期等
- **元数据提取**：提取文档的元数据，如发布日期、作者、机构等
- **结构化转换**：将非结构化数据转换为结构化格式，便于后续分析

#### 输出

| 输出类型 | 条件 | 说明 |
|---------|------|------|
| `CollectedData` | 信息收集成功 | 结构化的完整数据集 |
| `E010` (Warning) | 信息不足 | 带警告的数据集 |
| `E011-E015` (Error) | 数据源错误 | 数据获取失败 |

**CollectedData 结构**：

```typescript
interface CollectedData {
  metadata: {
    collectionTime: Date;
    dataSourceStats: Record<string, number>;  // 各源提取条数
    qualityScore: number;                    // 0-100 综合质量分

    warnings?: string[];
  };

  sources: {
    academic: Source[];
    industry: Source[];
    news: Source[];
    official: Source[];
    expert: Source[];
  };

  extractedInfo: {
    keyPoints: KeyPoint[];
    statistics: Statistic[];
    caseStudies: CaseStudy[];
    quotes: Quote[];
  };

  relationships: {
    conceptRelations: ConceptRelation[];
    sourceRelations: SourceRelation[];
  };
}

interface Source {
  id: string;
  title: string;
  author?: string;
  publicationDate: string;
  sourceType: string;
  credibility: 'high' | 'medium' | 'low';
  url?: string;
  abstract?: string;
  content?: string;
  keywords: string[];
}
```

#### 耗时预估

30-120 秒（取决于主题复杂度和数据源响应速度）

### Step 4: 分析处理阶段

```
┌─────────────────────────────────────────────┐
│            Step 4: 分析处理引擎               │
│                                             │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐      │
│  │ 4.1 主题  │→│ 4.2 来源 │→│ 4.3 内容 │      │
│  │ 分析     │ │ 分析     │ │ 分析     │      │
│  └─────────┘ └─────────┘ └─────────┘      │
│         ↓                                   │
│  ┌─────────┐ ┌─────────┐                   │
│  │ 4.4 共识 │→│ 4.5 争议 │                  │
│  │ 分析     │ │ 分析     │                  │
│  └─────────┘ └─────────┘                   │
│         ↓                                   │
│  ┌─────────────────────────┐              │
│  │   AnalysisReport        │              │
│  │   (分析结果)            │              │
│  └─────────────────────────┘              │
└─────────────────────────────────────────────┘
```

#### 输入

`CollectedData` （来自 Step 3 的收集数据）

#### 分析维度详解

##### 4.1 主题分析

- **主题分解**：将研究主题分解为多个子主题和维度
- **概念映射**：构建主题相关的概念图谱和关系网络
- **趋势分析**：识别主题的发展趋势和演变过程
- **重要性评估**：评估主题在相关领域的重要性和影响

##### 4.2 来源分析

- **来源可信度评估**：综合评估不同来源的可信度和权威性
- **来源多样性分析**：分析来源的多样性和代表性
- **来源时间分布**：分析来源的时间分布，评估信息的时效性
- **来源引用关系**：分析来源之间的引用关系，识别核心文献和观点

##### 4.3 内容分析

- **关键信息提取**：提取和整理关键信息和要点
- **数据统计分析**：对相关数据进行统计和分析
- **案例研究分析**：分析相关的案例研究和实践经验
- **专家观点综合**：综合专家的观点和建议

##### 4.4 共识分析

- **共识识别**：识别不同来源一致认同的观点和结论
- **共识强度评估**：评估共识的强度和普遍性
- **共识形成原因**：分析共识形成的原因和背景
- **共识的应用价值**：评估共识的应用价值和实践意义

##### 4.5 争议分析

- **争议识别**：识别存在分歧和争议的观点
- **争议焦点分析**：分析争议的焦点和核心问题
- **争议原因分析**：分析争议产生的原因和背景
- **争议的潜在影响**：评估争议的潜在影响和发展趋势

#### 输出

| 输出类型 | 条件 | 说明 |
|---------|------|------|
| `AnalysisReport` | 分析成功完成 | 包含多维度分析结果的完整报告 |
| `E020-E025` (Warning) | 部分分析失败 | 带警告的部分分析结果 |

**AnalysisReport 结构**：

```typescript
interface AnalysisReport {
  topicAnalysis: {
    subtopics: SubtopicAnalysis[];
    conceptMap: ConceptMap;
    trends: Trend[];
  };

  sourceAnalysis: {
    credibilityDistribution: Record<string, number>;
    timeDistribution: Record<string, number>;
    diversityScore: number;
    coreSources: Source[];
  };

  contentAnalysis: {
    keyPoints: KeyPointAnalysis[];
    statistics: StatisticalAnalysis[];
    caseStudies: CaseStudyAnalysis[];
    expertViews: ExpertView[];
  };

  consensusAnalysis: {
    consensusPoints: ConsensusPoint[];
    consensusStrength: number;
    consensusOrigins: string[];
  };

  debateAnalysis: {
    debatePoints: DebatePoint[];
    debateIntensity: number;
    debateOrigins: string[];
  };

  researchGaps: {
    identifiedGaps: ResearchGap[];
    potentialDirections: string[];
  };

  summary: {
    keyFindings: string[];
    overallAssessment: string;
    dataQualityNote: string;
  };
}
```

#### 耗时预估

60-180 秒（取决于数据量和分析深度）

### Step 5: 报告生成阶段

```
┌──────────────────────────────────────────┐

│                                          │
│  AnalysisReport                          │

│  │ 5.1 选择  │ ← 根据配置选择模板和格式   │

│       ↓                                  │
```

#### 输入

- `AnalysisReport` （来自 Step 4 的分析结果）
- `InternalConfig` （来自 Step 1 的配置）

#### 子步骤详解

##### 5.1 模板选择

根据 `InternalConfig` 中的 `template` 和 `detail_level` 选择对应的模板：

| 模板变体 | 详细程度 | 适用场景 |
|---------|---------|---------|
| `standard` | `summary` | 快速了解、决策参考 |
| `standard` | `standard` | 常规研究、报告撰写 |
| `standard` | `detailed` | 深度研究、详细报告 |
| `academic` | `detailed` | 学术研究、文献综述 |
| `industry` | `detailed` | 行业分析、市场研究 |
| `policy` | `detailed` | 政策研究、合规分析 |

##### 5.2 数据映射

将 `AnalysisReport` 的分析结果映射到模板的各个字段：

```yaml
mapping_rules:
  执行摘要:
    内容: ← analysisReport.summary.keyFindings
    要点: ← analysisReport.summary.overallAssessment

  关键发现:
    发现项: ← analysisReport.contentAnalysis.keyPoints
    来源: ← analysisReport.sourceAnalysis.coreSources

  详细分析:
    子主题: ← analysisReport.topicAnalysis.subtopics
    内容: ← analysisReport.contentAnalysis

  共识领域:
    共识点: ← analysisReport.consensusAnalysis.consensusPoints

  争议领域:
    争议点: ← analysisReport.debateAnalysis.debatePoints

  来源:
    来源列表: ← analysisReport.sourceAnalysis.coreSources

  研究空白:
    空白点: ← analysisReport.researchGaps.identifiedGaps
```

##### 5.3 内容填充

针对每个章节字段，动态生成具体内容：

- **文本段落**：基于分析数据生成叙述性文字
- **表格**：生成数据对比和统计表格
- **列表**：生成要点列表和分类信息
- **引用**：插入正确格式的引用标记

##### 5.4 格式优化

- **Markdown 格式化**：确保 Markdown 语法正确
- **标题层级**：保持标题层级的一致性
- **排版优化**：调整段落、列表和表格的排版
- **引用格式**：确保引用格式符合要求

##### 5.5 语言润色

- **专业性**：使用规范的专业语言
- **清晰性**：确保逻辑清晰，表达流畅
- **一致性**：保持术语和风格的一致性
- **准确性**：确保信息准确无误

| 输出类型 | 条件 | 说明 |
|---------|------|------|
| `DraftReport` | 生成成功 | 完整的草稿报告 |
| `E030-E034` (Error) | 生成失败 | 报告生成错误 |

30-120 秒（取决于详细程度和内容量）

#### 子步骤详解

##### 6.1 内容检查

- **准确性检查**：验证信息的准确性和一致性
- **引用检查**：确保所有引用都已正确标注
- **逻辑检查**：确保报告逻辑清晰，结构合理

- **格式规范**：检查 Markdown 或 JSON 格式是否规范
- **引用格式**：检查引用格式是否符合要求
- **排版检查**：检查排版是否美观、易读
- **语言检查**：检查语言是否专业、流畅

- **综合质量评分**：基于内容、格式和准确性给出综合评分
- **质量改进建议**：识别可能的改进点并提供建议
- **风险评估**：评估报告可能存在的风险和局限性

##### 6.4 最终输出

- **格式转换**：根据配置转换为指定的输出格式
- **元数据添加**：添加生成时间、版本等元数据
- **错误处理**：处理可能的格式转换错误

## 3. 异常路径汇总

| 错误类型 | 触发条件 | 处理策略 | 影响 |
|---------|---------|---------|------|
| 输入验证错误 | 参数缺失或无效 | 返回错误信息，指导用户修正 | 立即终止 |
| 信息收集错误 | 数据源不可用或信息不足 | 降级处理，使用可用信息 | 部分功能受限 |
| 分析处理错误 | 分析失败或数据质量问题 | 部分跳过，继续其他分析 | 分析深度受影响 |
| 报告生成错误 | 模板加载失败或格式错误 | 回退到默认模板，简化格式 | 报告质量受影响 |
| 系统错误 | 内存不足或超时 | 终止并返回错误信息 | 立即终止 |

## 5. 状态机说明

Deep Research 技能的执行过程可以视为一个状态机，包含以下状态：

| 状态 | 描述 | 转换条件 | 下一状态 |
|------|------|---------|---------|
| **初始化** | 接收请求，准备执行 | 参数解析成功 | 主题分析 |
| **主题分析** | 分析研究主题，制定搜索策略 | 主题分析成功 | 信息收集 |
| **信息收集** | 从多个来源收集信息 | 信息收集成功 | 分析处理 |
| **分析处理** | 分析收集的信息，识别共识和争议 | 分析处理成功 | 报告生成 |
| **报告生成** | 根据分析结果生成报告 | 报告生成成功 | 质量检查 |
| **质量检查** | 检查报告质量，进行最终优化 | 质量检查通过 | 完成 |
| **完成** | 返回最终报告 | - | - |
| **错误** | 处理过程中出现错误 | 错误处理完成 | - |

通过这个状态机，可以清晰地跟踪 Deep Research 技能的执行过程，及时发现和处理异常情况，确保研究任务的顺利完成。
